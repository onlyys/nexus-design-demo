5 大病灶

1.0 的根本错位：不是细节，是骨架。

B01

信息结构错配

用"一次性的文章"承载"持续演进的工作"。要么不更新（信息流断），要么发新 Topic（议题碎片化）。

B02

输入路径反人性

先写正文 → 再传附件 → AI 总结附件。逼用户把素材"翻译"成 Markdown——这层加工成本就是没人用的核心原因。

B03

AI 是贴皮，不是骨架

只在"上传后总结一次"出现一次，是功能点。对照奚丹要求的"AI 喂养层"，AI 含量远远不够。

B04

发布即死

文章 + 评论的消费模型。议题被锁定在发布那一刻，作者无法追加进展、读者无法订阅、AI 拿不到时间线。

B05

共识层完全没建：默认透明 / 不可二次转发 / HR 联动——1.0 一项都没有

"可见范围"只是个下拉框，全凭自觉。会议结论里的硬约束（默认 90% 透明、白名单不能由发布者自选、白名单内容不能再转发）在 1.0 没有产品层面的兑现。

改动 01 · 解决 B01

把"一篇 Topic"重构为 Topic → Event → Block。

TOPIC公益测试提效讨论
EVENT #1 · 2026-05-08公益 AI 自动化测试提效讨论
BLOCKtext · pptPreview · file
EVENT #2 · 2026-05-09测试工作流现状与改进计划
EVENT #3 · 2026-05-15测试工作流本周进展与重大突破

为什么是这三层

真实的"工作议题"天然就是三层：Topic 是议题本身，Event 是这件事的每一次演进，Block 是 Event 内的内容粒度。

为什么不能少一层

少了 Event，时间线就丢了——AI 无法回答"这件事最近怎么样"。Notion 的纯文档模型不解决"演进"问题。

为什么不能用平铺评论代替

评论是平的、无结构、无法承载 PPT/会议/文件。等于退回 1.0 的"附件式正文"。

改动 02 · 解决 B02

从一个 Markdown 框，到 15 种原生 Block。

BLOCK SYSTEM15

Block 化不是为了编辑体验。
是为 AI 而做的。

富文本是渲染层，没有"块"概念；AI 没法按粒度做摘要 / 检索 / 喂养。Block 化是必要前提。

15 种 Block 类型
text
h1 / h2 / h3
quote
bulletList
numberList
divider
image
file
link
todo
table
code
pptPreview
htmlPreview
+扩展
关键设计
  • pptPreview / htmlPreview 是原生预览，不是附件挂件——决定"用户愿不愿意把素材放进来"
  • 所有 Block 在 factory.ts 集中工厂化，新增 Block < 100 行代码
  • 每个 Block 独立可寻址，AI 可按 Block 粒度做摘要 / 引用 / 检索

REASONING · 反问到死

为什么是这套，不是别的？

Q1
能不能不大改，只在 1.0 上加 AI 增强？ 不行。1.0 的根本问题是信息结构错配——在错的结构上加 AI，等于把烂泥糊得更厚。AI 拿不到时间维度、拿不到 Block 粒度，再强的模型也救不回来。结构问题必须先解决。
Q2
能不能用 Notion / 飞书文档替代自研？ 三道硬墙：① 数据归属——AI 喂养层要求数据沉淀在 SSV / 腾讯内部；② 场景定制——Topic→Event→Block 是为"工作议题"定制的语义；③ 共识约束——默认透明、不可转发、HR 联动这些 SSV 特有要求，第三方做不到。
Q3
能不能只升级编辑器，发布后保持 1.0？ 不行。如果发布后还是文章模式，作者就没有"追加 Event"的入口，Event 结构在用户感知上就是死的。编辑态和消费态必须同构——这是产品一致性的基本要求。
Q4
6 月试运行最大风险是什么？ 不是工程量，是认知迁移。1.0 用户心智是"写一篇 Topic"，2.0 要求心智变成"积累一个议题"。需要高层示范（OKR 自上而下入库）+ 真实案例引导 + "录入=产出"的 HR 回报配套。
Q5
会不会"既要又要"——既要轻量录入又要结构化？ 不冲突。结构化是 AI 帮你做的，不是要求作者做的。作者只丢素材，结构化由系统自动生成 Event 草稿。这正是 2.0 的核心设计哲学：让结构化对用户透明。
